\section{Summary of Progress}
% ============================================================

We have implemented all six forward-pass CUDA kernels planned for this milestone: tiled GEMM, fused LayerNorm, online Softmax, GELU activation, cross-entropy loss, and Flash Attention. Each kernel is validated against a CPU reference and PyTorch ground truth, and a timing harness reports runtime, GFLOPS, and memory bandwidth. The host-side infrastructure (memory management, kernel launch wrappers, test suite, and plotting scripts) is complete.

Relative to the Milestone 2 design, the full forward-pass kernel suite is done. Backward passes, the Adam optimizer kernel, mixed precision, and multi-GPU training are not yet started and are targeted for Milestones 4 and 5.

\paragraph{Tiled GEMM.}
Each $32{\times}32$ thread block loads a corresponding sub-tile of $A$ and $B$ into shared memory and accumulates the partial dot-product there. At large problem sizes (e.g., $M{=}N{=}K{=}4096$), the algorithmic arithmetic intensity is $\approx 683$\,FLOP/byte, well above the ridge point ($\approx 24$\,FLOP/byte), making the kernel compute-bound. The planned next improvement is register tiling, where each thread computes a $4{\times}4$ output sub-tile and keeps accumulators in the register file throughout the $K$-loop, reducing synchronization overhead.

\paragraph{Flash Attention.}
Following \cite{dao2022}, we tile $Q$, $K$, $V$ into SRAM-sized blocks and maintain a running maximum $m^{(j)}$ and sum $\ell^{(j)}$, applying a correction factor $e^{m^{(j-1)}-m^{(j)}}$ when the maximum changes. The full $S{\times}S$ score matrix is never written to DRAM, reducing memory from $O(S^2)$ to $O(S{\cdot}D)$. Causal masking is applied by zeroing scores for keys at positions $> q$ before each softmax update.

\section{C++ Wrapper and CPU Code}
% ============================================================

Each kernel is driven by a thin host wrapper that allocates device buffers via \texttt{cudaMalloc}, transfers inputs with \texttt{cudaMemcpy}, launches the kernel, and copies results back. Timing uses a single CUDA event pair bracketing 10--20 launches (kernel-dependent) with one untimed warm-up, implemented in \texttt{tests/test\_utils.hpp}. A plain triple-loop C++ reference implementation exists for each operation and is used for correctness checks and, in the case of GEMM, for direct CPU-vs-GPU speedup comparison.

% ============================================================
\section{Correctness Testing}
% ============================================================

Each kernel is validated in two ways: element-wise comparison against the CPU reference, and comparison against PyTorch-generated \texttt{.bin} tensors (\texttt{scripts/generate\_ref\_data.py}). GEMM, Softmax, GELU, Cross-Entropy, and Flash Attention all pass. LayerNorm passes CPU comparison but fails the PyTorch relative-error threshold at large hidden dimensions ($H{=}768$). This is a test sensitivity issue, not an algorithmic bug: absolute error stays below $4{\times}10^{-7}$ across all cases, but the affine output $\gamma \cdot z + \beta$ can land near zero for some elements (with random $\gamma,\beta \in [-1,1]$), causing the relative error metric to spike. The tolerance threshold will be relaxed to reflect the actual floating-point accuracy.

The four memory-bound kernels are limited by the bandwidth ceiling. The main remaining lever is vectorized loads (\texttt{float4}) to reduce instruction overhead. GEMM and Flash Attention are compute-bound; the gap from the 16.3\,TFLOPS FP32 peak is primarily due to the single-output-per-thread tile strategy in GEMM (13.5\% of peak at $4096{\times}4096$). Register tiling and software pipelining (double-buffering with \texttt{cp.async}) are the planned next steps to close this gap.

\section{Challenges and Next Steps}
% ============================================================

\paragraph{Challenges.}
LayerNorm required two separate shared-memory reduction passes to avoid catastrophic cancellation in the variance computation; the algebraically equivalent one-pass formula $E[x^2]-E[x]^2$ is numerically unstable when the mean is large. Flash Attention required careful per-tile bookkeeping of the running-max correction factor when streaming $K$ and $V$ blocks.
