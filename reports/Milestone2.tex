\section*{1. Project Scope and Problem Definition}

We are building an end-to-end C++/CUDA training pipeline for a decoder-only GPT-2-style transformer~\cite{radford2019}: 6 layers, hidden dimension 512, 8 attention heads, context 256 tokens ($\sim$30M parameters), trained on WikiText-2 ($\sim$2M tokens, vocabulary 20{,}000) with BF16 mixed precision, FP32 master weights, and the Adam optimizer, distributed across multiple GPUs via MPI. Inputs are preprocessed token IDs; outputs are trained weights and a per-epoch training loss and validation perplexity log. We use a fixed-length context, word-level tokenization, and data parallelism only. A successful outcome means validation perplexity decreases over training, custom CUDA kernels achieve measurable hardware utilization, multi-GPU training scales efficiently, and a POS-augmented variant demonstrates task-level parallelism.

\section*{2. Survey of Related Work}

The GPT-2 architecture~\cite{radford2019} defines our model structure. For GEMM, we follow the shared-memory tiling and register-file strategy of~\cite{boehm2022}. Our custom BF16 batched GEMM \emph{extends} HW4 (FP32, single matrix) by operating in mixed precision with FP32 accumulation and handling the strided batched layout required for multi-head projections. For attention, we implement the full Flash Attention forward and backward~\cite{dao2022,dao2023}. This \emph{extends} HW6 (a forward-only standalone kernel) by integrating recomputation-based gradients into the training loop. Our MPI strategy is data-parallel gradient All-Reduce~\cite{li2020,sergeev2018}, a different pattern from the sequence-distributed attention in HW7. Compared with \texttt{llm.c}~\cite{karpathy2024}, we write all kernels from scratch and implement our own Ring All-Reduce rather than using cuBLAS/NCCL.

\section*{3. Computational Challenges}

\textbf{Numerical correctness at BF16.} The reduced mantissa (8 bits vs.\ 24 for FP32) makes accumulation errors in GEMM and softmax significant. We accumulate in FP32 within each kernel and validate against PyTorch, accepting relative error below $10^{-2}$.

\textbf{Backward pass and communication overlap.} A full transformer backward involves six GEMMs per layer plus the Flash Attention backward with recomputation, requiring careful management of transposed weights and residual connections. Overlapping the All-Reduce with the backward pass adds further complexity: reductions must run on a separate CUDA stream and synchronize only when the optimizer needs the averaged gradient. We validate gradients layer-by-layer against PyTorch autograd.

\section*{4. CUDA Implementation Plan}

\textbf{BF16 batched GEMM.} All linear layers (Q/K/V projections, output projection, FFN) are tiled into shared memory, accumulated in FP32 registers, and written as BF16 outputs.

\textbf{Flash Attention forward and backward.} The forward kernel tiles over sequence length with online softmax in SRAM; the backward recomputes attention weights from saved statistics and produces gradients for Q, K, V.

\textbf{Fused LayerNorm / GELU / residual.} These individually memory-bound operations are fused into single kernels to eliminate redundant global memory traffic.

\textbf{Fused cross-entropy + Adam.} The loss and softmax gradient are fused to avoid materializing probabilities; the Adam kernel updates FP32 master weights from BF16 gradients and moment buffers in one pass.

All kernels are validated against PyTorch before optimization.

\section*{5. MPI Implementation Plan}

Each MPI rank owns a full model copy and a disjoint data shard. Per step: (i) each rank runs forward/backward on its mini-batch; (ii) gradients are averaged via Ring All-Reduce; (iii) each rank applies the optimizer. The Ring All-Reduce uses \texttt{MPI\_Isend}/\texttt{MPI\_Irecv} with CUDA-aware MPI for direct GPU-to-GPU transfers; small tensors are bucketed to amortize startup latency. The All-Reduce of layer $\ell$'s gradients is overlapped with the backward pass of earlier layers on a separate CUDA stream.

For the POS comparison, \texttt{MPI\_Comm\_split} partitions the communicator into two independent sub-groups running separate data-parallel training runs with no cross-group communication.

\section*{6. Profiling and Performance Analysis Plan}

\textbf{Kernel Roofline.} Measure achieved FLOPS and memory bandwidth for each major kernel (batched GEMM, Flash Attention fwd/bwd, fused LayerNorm) via CUDA events and \texttt{ncu}; plot on a Roofline diagram to classify compute- vs.\ memory-bound behavior.

\textbf{Scaling.} Strong scaling: fix total batch size, vary GPU count, measure efficiency $E = S(P)/P$. Weak scaling: scale per-GPU batch proportionally and check for constant step time.

\textbf{Communication cost.} Time Ring All-Reduce for varying message sizes; fit $\alpha + \beta n$; compare against theoretical Ring and Tree lower bounds.

\textbf{Task vs.\ data parallelism.} With 4 GPUs: both POS configs simultaneously on 2 GPUs each vs.\ each config sequentially on 4 GPUs; compare total wall-clock time.
